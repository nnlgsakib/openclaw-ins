# Phase 9: Channel Pairing Flows - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement channel-specific setup wizards that activate when a user clicks "Connect" on a channel card (Phase 8). Each channel type has a dedicated pairing flow: WhatsApp uses QR code scanning, Telegram and Discord use guided token entry with validation. This completes the connection story started in Phase 8.

</domain>

<decisions>
## Implementation Decisions

### Pairing Flow Architecture
- Pairing wizards rendered as modal dialogs (not separate pages/routes)
- Modal triggered from ChannelCard's "Connect" button in Phase 8
- Each channel type has its own pairing component inside the modal
- Modal state managed locally in channels.tsx with useState<ChannelType | null>

### WhatsApp QR Code Pairing
- QR code generated by OpenClaw backend, fetched as base64 image via Tauri command
- `get_whatsapp_qr` Tauri command returns QR code data URL
- Display QR code image with instructions ("Scan with WhatsApp on your phone")
- Poll for connection status (every 5s) while QR is displayed
- QR refreshes automatically when expired (OpenClaw regenerates)
- Timeout handling: show "QR expired, generating new one..." message

### Telegram Bot Token Entry
- Input field for bot token (from @BotFather)
- Token validation via `validate_telegram_token` Tauri command
- Validates token format (numeric:alphanumeric pattern) before sending to OpenClaw
- Success/failure feedback with clear error messages
- Link to external guide: "How to create a Telegram bot"

### Discord Bot Token Entry
- Input field for bot token (from Discord Developer Portal)
- Token validation via `validate_discord_token` Tauri command
- Similar validation pattern to Telegram
- Link to external guide: "How to create a Discord bot"

### Shared Pairing Components
- `PairingModal` — wrapper with Dialog, title, close button
- `QrPairingStep` — QR display with loading/polling states
- `TokenPairingStep` — token input with validation
- Reuse shadcn/ui Button, Input (from existing components or install if needed)

### the agent's Discretion
- Modal animation style (fade vs slide)
- Exact QR code display size and styling
- Token input masking (show/hide toggle)
- Validation timing (on blur vs on submit)
- Whether to install shadcn/ui Input component or use native input with Tailwind

</decisions>

## Specific Ideas

- "Token entry should validate before submitting — don't waste the user's time with server round-trips for obvious format errors"
- "Show a link to setup guides for each platform — users might not know where to get tokens"
